# Model Catalog Auto-Update — Periodic models.dev Refresh

**Status:** COMPLETED 2026-07-18 — see [implementation plan](./2026-07-18-model-catalog-periodic-refresh_plan_completed.md) (one live check deferred to the [livetest doc](./2026-07-18-model-catalog-periodic-refresh_livetest.md))
**Decisions:** A = 6h · B = regenerate snapshot (in scope) · C = curation deferred
**Date:** 2026-07-18
**Author:** Claude (Opus 4.8) with James
**Trigger:** "Why is Sonnet 5 not here? I thought we were auto-updating model lists now?"

---

## TL;DR

Sonnet 5 is missing **not** because Claude lacks an auto-discovery path — the path
exists and is fully wired. It is missing because the one discovery source for Claude
models (**models.dev**) is fetched **only once, at app startup**, with no periodic
refresh. A long-running app instance (this app routinely runs for days) started before
`claude-sonnet-5` was published upstream and therefore never re-fetched it. The
committed offline fallback snapshot is also stale (stops at Sonnet 4.6).

The original framing — "wire a Claude discovery service" — is based on a wrong premise
and would duplicate existing, working infrastructure. The real fix is a small
periodic-refresh timer plus two optional quality follow-ups.

---

## Verified root-cause evidence

All claims below were confirmed by reading the executing code path and by live-fetching
models.dev on 2026-07-18.

1. **The Claude CLI cannot list models.** `claude` v2.1.214 exposes only `--model`;
   there is no `--list-models` / `models` subcommand. So — unlike Codex, Cursor, and
   Copilot — there is no CLI to poll for Claude. (Verified by running `claude --help`.)

2. **models.dev IS the Claude discovery source, and it is wired end-to-end.**
   - `ModelsDevService` fetches `https://models.dev/api.json`, parses the full
     `anthropic/*` model list, and calls `getUnifiedModelCatalog().onModelsDevRefreshed()`
     (`src/main/providers/models-dev-service.ts:162-174`, `:187-189`).
   - The unified catalog maps the `anthropic` namespace → `claude`
     (`src/main/providers/unified-model-catalog-normalizers.ts:88-90`) and adds
     models.dev-only entries into the `claude` bucket
     (`src/main/providers/unified-model-catalog-service.ts:402-439`).
   - The catalog pushes `models:catalog-updated` over IPC; the renderer
     `UnifiedCatalogStore` refreshes the picker live
     (`provider-handlers.ts:443-466`, `provider.preload.ts:307-344`,
     `unified-catalog.store.ts`).
   - The picker (`CompactModelPickerComponent`) reads `displayModelsForProvider()`
     from the store, falling back to the static list only until the catalog loads.
     models.dev-only ids are **not** filtered — they are humanised into a display name
     (`unified-catalog.store.ts:95-116`).

   → If `claude-sonnet-5` were in the running catalog, the picker **would** show it.

3. **Live models.dev already lists `claude-sonnet-5`.** Live fetch on 2026-07-18
   returned 14 anthropic models including `claude-sonnet-5`, `claude-opus-4-8`,
   `claude-opus-4-1`, `claude-opus-4-5-20251101`, etc.

4. **The refresh is one-shot — this is the bug.** `ModelsDevService.refresh()` is
   invoked exactly twice, both at startup:
   - `src/main/bootstrap/infrastructure-bootstrap.ts:80`
   - `src/main/app/unified-model-catalog-initialization.ts:123`

   There is **no `setInterval`** re-invoking it. The 6h TTL inside `refresh()` only
   *guards against over-fetching*; nothing schedules a second call. Compare:
   - `codex-cli-discovery-service.ts:40` — `setInterval` polling
   - `cursor-copilot-cli-discovery-service.ts:62` — `setInterval` polling
   - `catalog-override-source.ts:412` — `setInterval` (6h) for the remote override

   models.dev is the odd one out: startup-only. A long-running instance never picks up
   newly-published models until it restarts.

5. **The committed offline snapshot is stale.** `models-dev-snapshot.generated.ts`
   contains anthropic models only up to `claude-sonnet-4-6` — no `claude-sonnet-5`. So
   if the startup live fetch fails (6s timeout / offline), the fallback has no Sonnet 5
   either. The snapshot is regenerated by `npm run sync:model-catalog` and has not been
   run since Sonnet 5 landed.

6. **UX-quality gap (cosmetic, not the blocker).** A models.dev-only Claude entry is
   added with `tier: 'balanced'`, no `family`, no `pinned`, and no curated `name`
   (`unified-model-catalog-service.ts:427-437`). So Sonnet 5, once discovered, would
   appear as a humanised "Claude Sonnet 5" in a default group rather than as a polished
   "Sonnet 5" inside the Sonnet family near the top. It would still be selectable.

## Proposed scope

### Item 1 — Periodic models.dev refresh (CORE fix — required)

Add a recurring refresh so a running instance picks up newly-published models without a
restart, mirroring the existing codex/cursor/override pollers.

- Give `ModelsDevService` a `start()` / `stop()` pair that runs `refresh()` immediately
  and then on a `setInterval` (proposed 6h, matching the existing TTL and the override
  source; timer `unref()`'d so it never blocks exit), or add an equivalent scheduler in
  `unified-model-catalog-initialization.ts`. Prefer the `start()/stop()` shape for
  parity with the other discovery services and a clean bootstrap teardown hook.
- Coalesce with the existing in-flight/TTL guard so the extra startup call is a no-op.
- Wire `stop()` into bootstrap teardown.
- Tests: interval fires refresh; TTL/in-flight coalescing holds; teardown clears timer.

**Outcome:** Sonnet 5 (and every future model models.dev publishes) appears within one
refresh interval, no restart needed. This alone fixes the reported problem for any
app that can reach models.dev.

### Item 2 — Regenerate the committed offline snapshot (quality — recommended)

Run `npm run sync:model-catalog` to refresh `models-dev-snapshot.generated.ts` so fresh
installs and offline/first-launch (before the first live fetch lands) already know the
current model set, including Sonnet 5. Low-risk, generator-owned file.

### Item 3 — Curate discovered Claude entries (quality — optional)

Give models.dev-only Claude models correct `tier`/`family`/display-name so they slot
into the picker's Sonnet/Opus/Haiku groups instead of a humanised default row. Options:
derive `family` from the id (`sonnet`/`opus`/`haiku`/`fable`) and map tier accordingly,
either in the catalog rebuild or via the existing static-metadata overlay. Purely
cosmetic; can be deferred.

## Non-goals

- No new "Claude discovery service" (the models.dev path already covers Claude).
- No hand-editing new model IDs into `PROVIDER_MODEL_LIST` as the primary mechanism
  (that is the static fallback, not the auto-update path).
- No change to pricing/context-window overlay behaviour.
